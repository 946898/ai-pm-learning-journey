## 一、Agent核心循环：Perceive→Think→Act→Observe

```markdown
Perceive（感知）→ Think（思考）→ Act（行动）→ Observe（观察）
          ↑                                          |
          └──────────────────────────────────────────┘
```

| 步骤     | Agent做了什么                              |
| -------- | ------------------------------------------ |
| Perceive | 接收到用户指令：“帮我翻译这个PDF”          |
| Think    | 思考：我需要先读取PDF内容，然后翻译        |
| Act      | 调用PDF读取工具，获取文本                  |
| Oberseve | 拿到了 PDF 的文字内容                      |
| Think    | 思考：内容拿到了，开始翻译                 |
| Act      | 执行翻译                                   |
| Oberseve | 翻译完成                                   |
| Think    | 思考：翻译结果是否符合用户预期？可以返回了 |
| Act      | 将翻译结果返回给用户                       |

### 1.1Think缺失导致死循环

> Agent陷入死循环可能是Think步骤出现问题，不一定是加个重试上限。

```erlang
翻译 PDF → 拿到结果 → 没有思考"是否完成" → 又去翻译 → 又拿到结果 → ...
```

**根本原因是Think步骤缺失**，Think决定“做什么”，重试上限决定“做多久”。两者缺一不可。

## 二、Function Calling与MCP

> 解释循环中的Act步骤，Agent如何Work

### 2.1Function Calling：LLM的“表达方式”

LLM输出一段结构化的JSON，告诉宿主程序想要调用的工具。

```json
{
  "tool": "get_weather",
  "parameters": {
    "city": "上海"
  }
}
```

### 2.2MCP：宿主侧的“连接协议”

MCP是宿主程序（如Claude Code）用来管理和连接外部工具服务器的协议。

解决的是“工具从哪来、怎么调”的问题。

```sql
LLM 输出 Function Call → Host 解析 → 通过 MCP（或其他方式）路由到具体工具 → 执行 → 结果返回给 LLM
```

### 2.3它们的关系

Function Calling是LLM的嘴（表达意图），MCP是Host的手（连接和执行工具）。

不论工具为内置或MCP配置，LLM均输出同样格式JSON的Function Calling。

### 2.4工具描述的质量决定一切

> 记住：工具描述不是写个人看的注释，而是写给LLM看的使用说明书。

```vb
// ❌ 模糊的描述——处理数据？什么数据？怎么处理？
name: "data_processor"
description: "处理数据"

// ✅ 精准的描述——知道遇到CSV转JSON的需求时刻调用它
name: "csv_to_json_converter"
description: "将 CSV 格式的文本转换为 JSON 数组，
每一行成为数组中的一个对象，首行作为字段名"
```

### 2.5三层安全设计

#### 比如“发邮件”：

1、工具描述的约束，调用前必须和用户确认内容

2、LLM Think评估，在LLM或Think步骤判断“这个操作是否需要确认？”

3、Host拦截，宿主程序对高风险操作弹出确认提示

## 三、规划与任务分解：Agent如何大拆分

**规划**：列出需求点、确定技术栈、规划页面结构、拆除任务清单。

### 3.1带依赖关系的拆解

除了TODO List，还要有依赖关系。

```yaml
任务 1: 初始化项目        → 无依赖
任务 2: 搭建页面布局      → 依赖任务 1
任务 3: 实现暗黑模式      → 依赖任务 2
任务 4: 写单元测试         → 依赖任务 2、3

```

依赖关系，方便：

1、Agent知道执行顺序

2、保证依赖任务完成后才执行下游任务

3、多Agent模式下哪些任务可以并行分配

### 3.2两种执行策略

| 策略                      | 说明                                     | 适用场景             |
| ------------------------- | ---------------------------------------- | -------------------- |
| **Plan-then-Execute**     | 先规划好所有步骤，然后逐步执行           | 需求明确、步骤可预见 |
| **Interleaved（交替式）** | 规划一步、执行一步、再根据结果规划下一步 | 不确定性高的任务     |

Agent需要重新审视整个计划，调整方案再继续执行。

> 🚨 **还有一个容易忽略的点：探索先于规划！**
>
> 比如要重构一个认证模块，Agent 不应该上来就写计划。而是先审视一下项目现有的认证模块——看能不能改、改了影响多大——然后再做规划。

## 四、记忆系统：短期、长期、工作记忆

### 4.1三种记忆

| 类型         | 类比                   | Agent 中的体现           | 生命周期       |
| ------------ | ---------------------- | ------------------------ | -------------- |
| **短期记忆** | 和朋友聊天时的对话内容 | 当前对话的上下文         | 本次对话       |
| **长期记忆** | 手机备忘录/**收藏**    | MEMORY.md、CLAUDE.md     | 跨对话保留     |
| **工作记忆** | 做数学题时的草稿纸     | TodoWrite 任务清单、Plan | 任务结束就丢弃 |

> 需要跨对话保留的→长期记忆。只在当前任务中有用的→工作记忆。
>
> 比如在20日沟通了，23日再来沟通，这时候需要获取20日的对话信息，这个就是需要长期记忆。而工作记忆已经遗忘了。

## 五、上下文窗口管理：Agent“内存”管理

如同手机的16G+1T，前者是运行内存，所谓内存；后者是存储空间，所谓硬盘。

LLM的上下文窗口是有限制的，是内存。处理大型项目的Agent不能把所有文件塞进上下文内。因此需要制定加载策略：

### 5.1按需加载

先搜索定位，再精确读取。

比如Claude Code修复BUG，做法是：①先用grep搜索关键词，找到3个相关文件；②只读取这3个文件的内容；③修改完成。

### 5.2摘要压缩

对话过长时，将早期对话内容压缩成摘要，保留关键信息，释放上下文空间。

### 5.3子Agent分担

将任务拆分给子agent处理，每个子agent有自己独立的上下文窗口。将一个大的上下文需求分散到多个小窗口内。

+记忆互通？

### 5.4避免冗余

已知信息不需重复加载。

> - 避免冗余：**不加**载已知重复信息
> - 摘要压缩：将已知信息压缩**变短**

## 六、ReAct范式：边想边做的艺术

ReAct = Reason + Act

> Perceive→think→act→observe是通用的设计模式
>
> ReAct是具体的实现方法

ReAct之前的两种做法，ReAct就是知行合一，将thought和action交替进行。

| 做法                                 | 问题                                     |
| ------------------------------------ | ---------------------------------------- |
| **只推理不行动**（Chain-of-Thought） | LLM 一直在"想"，但不调工具，容易产生幻觉 |
| **只行动不推理**（Action-only）      | 直接调工具，不说明为什么，遇到意外就懵了 |

```vb
Thought: 我需要先确认《三体》的作者，不能凭记忆猜
Action: search("三体 作者")
Observation: 刘慈欣

Thought: 确认是刘慈欣，接下来搜他的其他作品
Action: search("刘慈欣 其他作品")
Observation: 《球状闪电》《流浪地球》《超新星纪元》...

```

### 6.1ReAct的两大核心价值（应对不确定性）

#### 6.1.1自我纠错

```vb
Thought: 我需要先确认作者
Action: search("三体 作者")
Observation: 没有找到相关结果
Thought: 搜索失败了，可能是关键词不对，换个方式试试
Action: search("《三体》 科幻小说 作者是谁")
Observation: 刘慈欣

```

如果是Action-Only或者Think-Only，搜索失败后Agent直接跳下一步，然后回绝你了。

#### 6.1.2可追溯性

> 类似深度思考模式的思考过程

每一步推理都显示输出，开发者能看到Agent在想啥。出了问题可以快速定位。

Agent在每次调用工具前输出的分析和推理文字，就是ReAct中Thought的体现。

#### 6.1.3不需要ReAct场景

- 流程固定

- 没有不确定性的任务

比如"每天早上查天气 → 格式化 → 发通知"，这种固定流水线甚至不需要 LLM 来编排，一个定时脚本就够了。ReAct 的价值在于**应对不确定性**，固定流程用它反而是浪费 token 。

## 七、多Agent协作：一个人干不完叫帮手

### 7.1需要多Agent的理由

| 优势           | 说明                                            |
| -------------- | ----------------------------------------------- |
| **并行效率**   | 没有依赖的任务可以同时跑                        |
| **上下文隔离** | 每个 Agent 只加载自己需要的信息，不互相污染     |
| **专业化**     | 每个 Agent 专注一个领域，配备针对性的工具和提示 |

### 7.2两种协作架构

#### 7.2.1中心编排（Orchestrator）：

```css
        [主 Agent]
       /    |     \
  后端Agent  前端Agent  测试Agent

```

主Agent分配任务、收集结果、协调信息。子Agent之间不直接通信。

#### 7.2.2去中心化（Peer to Peer）：

```markdown
  后端Agent ←→ 前端Agent
      ↕            ↕
     测试Agent ←→ ...


```

Agent之间直接通信，通过共享状态来同步。

### 7.3Claude Code的协作架构

**中心编排。** Claude Code 有一个主 Agent 做全局把控，spawn 出子 Agent 来并行处理任务。子 Agent 之间不直接通信，所有信息流经主 Agent。

为什么选这种？因为主 Agent 能**统一调度**——当后端 Agent 改了 API 字段名，主 Agent 可以通知前端 Agent 和测试 Agent 同步修改。如果是去中心化的，这个信息同步就会很混乱。

### 7.4中心编排的代价

主 Agent 会成为**瓶颈**——所有信息都要经过它，带来额外的 token 消耗和延迟。所以实际使用中，主 Agent 不会让子 Agent 汇报每一个细节，而是只收集**关键结果和变更**。

## 八、错误处理与自我纠正：Agent犯错了怎么办？

> 关键是如何发现错误和处理错误

### 8.1三类错误信号来源

| 来源         | 说明                                                    | 可靠性       |
| ------------ | ------------------------------------------------------- | ------------ |
| **环境反馈** | 执行结果直接告诉你错了（测试失败、命令报错、API 404）   | ⭐⭐⭐ 最高     |
| **人类反馈** | 用户主动指出问题（"你改错文件了"）                      | ⭐⭐           |
| **自我反思** | Agent 在 Thought 中自己发现不对（"这个文档版本不匹配"） | ⭐ 可能反思错 |

> 需要注意的是，这里的可靠性指的是**信号的确定性**——环境反馈是程序化的确定结果， 而人类反馈可能存在表述模糊的情况。但在**意图判断**层面，人类反馈的权威性是最高的。

### 8.2四种应对策略

| 策略           | 说明                                 | 适用场景                   |
| -------------- | ------------------------------------ | -------------------------- |
| **换方式重试** | 换关键词、换思路，不是重复同样的操作 | 第一次失败，可能是方法不对 |
| **回退重来**   | 撤销修改，从头用完全不同的方案       | 连续修补越改越乱           |
| **升级求助**   | 请求用户介入，或调用更强的模型       | 自身能力不足               |
| **优雅失败**   | 承认失败，报告已尝试的方案           | 所有策略都用尽             |

> Think决定做什么，重试上限决定做多久。没有Think，Agent会盲目重试同样的操作；没有重试上限，即使有Think也可能无限尝试。两者缺一不可！

实际中，一个错误可能需要**组合使用**多种策略。比如执行 `npm install` 报磁盘空间不足，Agent 可能需要先清理缓存（换方式重试），但清理磁盘属于有风险的操作，又需要先问用户（升级求助）。

## 九、安全与对齐：给Agent装刹车

Agent能力越强，越需要安全约束。不可选，必须的！

### 9.1风险分级权限模型

| 风险等级     | 操作类型                      | 处理方式                                 |
| ------------ | ----------------------------- | ---------------------------------------- |
| **无风险**   | 读取文件、搜索                | 直接执行，不问                           |
| **中等风险** | 编辑文件                      | 可配置（严格模式要问，宽松模式自动执行） |
| **高风险**   | Bash 命令、删除文件、推送代码 | 默认必须确认                             |
| **禁止**     | 恶意代码、攻击系统            | 无论如何都不执行                         |

### 9.2四层防御

| 层级                | 机制                 | 说明                 |
| ------------------- | -------------------- | -------------------- |
| **第 1 层：模型层** | LLM 训练时内置的对齐 | 拒绝明显有害的请求   |
| **第 2 层：工具层** | 工具描述中的约束规则 | "发邮件前必须确认"   |
| **第 3 层：系统层** | 权限分级、沙箱隔离   | 限制可访问的目录范围 |
| **第 4 层：人类层** | Human-in-the-Loop    | 关键操作让用户确认   |

> 核心原则是：**每一层都不能完全信任上一层**

即使模型层没拦住（被 prompt injection 绕过了），工具层可以拦；工具层也没拦住，系统层可以拦；系统层也漏了，还有人类做最后把关。

## 十、编排框架与实战选型：用什么工具来造Agent

### 10.1三个层级

| 层级           | Agent 领域        | 前端类比   | 特点                         |
| -------------- | ----------------- | ---------- | ---------------------------- |
| **底层 SDK**   | Claude Agent SDK  | 原生 JS    | 最大灵活性，什么都能控制     |
| **编排框架**   | LangGraph、CrewAI | React、Vue | 核心抽象开箱即用，有约定模式 |
| **低代码平台** | Dify、Coze        | Webflow    | 可视化搭建，上手快但灵活性低 |

### 10.2选型指南

#### 10.2.1选底层SDK

需要极致控制力，团队技术能力强。比如 Anthropic 的 Claude Code，它的核心循环、上下文管理、权限系统、hook 机制都是从底层直接构建的——这种高度定制的场景，就适合在 SDK 层级或更底层去实现。

#### 10.2.2选编排框架

有开发能力的团队，想利用框架的核心抽象（Agent、Tool、Memory）快速搭建，同时保留一定的灵活性和可扩展性。

#### 10.2.3选低代码平台

不懂开发的人员也能用可视化方式搭工作流，或者需要快速验证想法。比如一个运营同学想搭个"每天自动总结新闻发到飞书群"的工作流，用 Dify 拖拖拽拽就行了。

> 💡 **一个有趣的趋势：**
>
> 当前 Agent 领域还在快速演化，**底层 SDK 的使用率在上升**。因为很多编排框架的抽象层反而成了限制——Agent 的行为很难被预定义的流程图完全覆盖。Anthropic 官方推荐的做法就是用轻量 SDK + 简单代码编排，而不是引入重型框架。
>
> 这就像前端早期——很多人觉得必须用框架，后来发现对于很多场景，原生 JS + 轻量库反而更合适 

## Conclude

- **核心循环**（Perceice→Think→Action→Observe）是 Agent 的"心跳"
- **工具调用**让它能"动手"
- **规划**让它能"拆解问题"
- **记忆**让它能"记住你"
- **上下文管理**让它能"高效工作"
- **ReAct**让它能"边想边做"
- **多 Agent**让它能"叫帮手"
- **错误处理**让它能"从失败中恢复"
- **安全与对齐**给它装上"刹车"
- **编排框架**提供"开箱即用的工具箱"
