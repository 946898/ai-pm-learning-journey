## 一、Conclude

1. Context Window组成部分是**【历史会话+当前输入+模型输出】**≈短期记忆+长期记忆（如在范围内）
2. **【每个模型的Context window并不一致】**，比如GPT-4o则是128k
3. 描述模型的**【128k的k不是存储大小单位】**，是数量单位**【1k===1000token】**
4. **【Context window并不是越大越好】**，越大代表计算复杂度越高，并且目标内容更难命中
5. 当context window**超过指定存储限制**的时候（如：128k），一些模型**会舍弃**早期的存储内容，还有一些**会直接报错**
6. **【128k的context window换算粗估成存储空间约为0.5mb大小】**

### 1.1为什么说历史会话+当前输入+模型输出不等于历史会话+当前会话？

#### 1.1.1发送时刻的组成（静态视角）

当我们在聊天框按下回车，发送给大模型（LLM）的 Prompt（提示词）里，包含且仅包含以下三块：

1. **系统提示词（System Prompt）**：也就是我们上一轮提到的 **Rules、AGENTS.md** 等硬性指令。它在后台，我们看不到，但占窗口空间。
2. **历史会话（History）**：**已经结束的、完整的**一问一答（之前的 User 输入 + Assistant 输出）。
3. **当前输入（Current Input）**：我们刚打完的这一句话。

由于AI在我们发送时，回答还不存在，没开始生成，所以模型输出这个点不对。

#### 1.1.2生成过程中的组成（动态视角）

当我们按下回车后，AI 开始一个字一个字地吐回答。此时，上下文窗口在**物理上**发生了变化：

- **它变成了：系统提示词 + 历史会话 + 当前用户输入 + 模型已输出的内容（实时追加）**
- **关键点**：AI 每生成 1 个 Token（词元），这个 Token 就被**立刻塞进上下文窗口的末尾**，作为下一次生成下一个 Token 的“依据”。
- 当 AI 终于回答完毕，它生成的所有内容就**固化**下来，成为了下一条消息的 **“历史会话”**。

#### 1.1.3为什么必须严格区分”历史“和”当前输出“？（工程视角）

如果我们在做产品（尤其涉及 Cursor 的异步索引或 OpenClaw 的记忆压缩），搞混这两个概念会引发严重的**成本灾难**：

- **“当前输出”是动态累加的**：如果你问 AI“写一个贪吃蛇游戏”，AI 生成了 2000 行代码。这 2000 行在生成过程中**全都要算在上下文窗口里**。如果窗口上限是 128K，生成到一半把窗口塞满了，AI 就会“断片”，忘记最开始的游戏规则，导致代码烂尾。
- **“历史会话”是需要压缩的**：正因为 AI 的输出会迅速占满窗口，所以你上一轮提到的**“压缩（Compression）”**和**“遗忘（Forgetting）”**才尤为重要。产品必须把那些已经聊了 50 轮的“历史会话”摘出来，用 500 字总结掉（摘要），然后丢弃原始对话，腾出空间给“当前输出”继续生成。

#### 1.1.4不能忽略的知识点

在“历史会话”和“当前输出”之外，还有一个占窗口的大户，也就是上一轮提过的 **Worktree（工作区快照）** 或 **代码上下文**。

- 在 Cursor 里，我们问 AI 一个问题，系统会先把**当前打开的文件内容**、**光标附近的代码**、以及**报错堆栈信息**全部塞进上下文窗口。
- 这些东西既不属于“历史会话”，也不属于“当前会话”，但它们可能吃掉 60% 的窗口空间。

#### 1.1.5产品设计理解

我们可以把上下文窗口想象成一个“沙漏”：

- 上半部分（固定位）：系统提示词+压缩后的历史摘要+当前打开的文件内容（context）
- 中间（可变）：当前用户的输入（Input）
- 下半部分（动态填充）：AI正在输出的回答（Output），它会像沙子一样不断留下来，直到塞满整个沙漏。

> **上下文窗口 = 系统指令（固定） + 历史摘要/快照（可被压缩） + 当前输入（触发词） + 实时生成的输出流（动态增长）。**

设计产品时，务必把“历史会话”看作可以被工程手段裁剪（截断/摘要）的对象。把“当前输出”看作必须时刻监控其长度的活动进程。

### 1.2在设计“记忆压缩“和”遗忘“策略做决策要点

#### 1.2.1上下文窗口和长期记忆的比喻

> - **上下文窗口（Context Window） = 电脑的“内存（RAM）”**
>   它存放的是**当前正在运行、正在计算**的所有东西。速度快，但容量有限（如 128K tokens），关机（会话结束）或断电（超出长度）就没了。
> - **长期记忆（Long-term Memory） = 电脑的“硬盘（SSD）”**
>   它存放的是**永久或半永久**的数据。容量大（向量库、SQLite 数据库），但访问慢，需要专门的检索（Query）才能调取到内存里。

#### 1.2.2误区一：“超出上下文窗口的，自动变成长期记忆”

- **真相**：超出上下文窗口的旧对话，**绝大部分会被直接丢弃（遗忘）**，而不是自动存入长期记忆。
- **举个例子**：假设窗口是 10 句话，你聊到第 11 句话时，第 1 句话被挤出去了。**第 1 句话既不在窗口里，也不在向量库里**——它就像空气中的水蒸气，彻底消失了，除非你的系统在它消失前**主动**执行了“存档”操作。
- **对产品的启示**：你必须显式地写代码（比如调用 Embedding 接口存入 LanceDB，或调用 SQLite 写入 FTS5），才能把“超出窗口”的数据变成“长期记忆”。**<u>没有存储动作，就没有长期记忆。</u>**

#### 1.2.3误区二：“上下文窗口内只有短期记忆”

上下文窗口里装的**不仅仅是**你刚刚说的几句话（短期记忆），它还包括三大类东西，其中很多都属于“长期记忆的副本”：

1. **系统级长期规则（System Rules）**：比如“禁止用 eval()”。这是从长期配置里硬编码加载进来的，虽然占窗口，但它是永久性指令，不是短期会话。
2. **检索到的外部知识（RAG 结果）**：当你问 AI 问题时，后台会去 LanceDB 或 SQLite 里把相关文档搜出来，**强行塞进当前的上下文窗口**。此时，长期记忆的“副本”正躺在窗口里被 AI 阅读。
3. **正在生成的当前输出（Current Output）**：这确实是短期且瞬时的。
4. **近期历史会话（Recent History）**：这才是你理解的“短期记忆”。

**结论**：上下文窗口更像是一个**“混合工作台”**，台面上既有从硬盘拿过来的长期资料（Rules、RAG），也有正在临时产生的便签纸（当前对话）。所以不能说窗口里只有短期记忆。

#### 1.2.4正确的三层“记忆体系”认知

| 层级   | 名称                        | 容量              | 访问速度       | 典型内容                                                     | 是否可修改         |
| :----- | :-------------------------- | :---------------- | :------------- | :----------------------------------------------------------- | :----------------- |
| **L1** | **当前上下文窗口**          | 极有限（如 128K） | 极快（毫秒级） | 当前问题 + 即时回答 + 检索到的资料块                         | 动态追加，不可回退 |
| **L2** | **压缩摘要（工作记忆）**    | 中等（几千字）    | 中等           | **上一轮你问的“压缩”产物**：把 50 轮对话浓缩成 500 字，仍放在窗口里 | 每次对话后重写     |
| **L3** | **外部长期存储（向量/DB）** | 无限（理论上）    | 慢（需检索）   | 历史完整对话文件、代码库索引、用户偏好                       | 可增删改查（CRUD） |

#### 1.2.5产品设计上的最终结论

1. **上下文窗口绝对不等于短期记忆**。它是“当前正在使用的所有相关数据的集合”，你得把它当作**开销极大的计算资源**来管理。
2. **超出窗口的数据，默认等于“被遗忘”，而非“被存档”**。因此，你必须在数据被挤出窗口前，启动你上一轮提到的 **“写入门槛（Write Gate）”** 和 **“压缩（Compression）”**，主动把窗口里的精华提炼出来存进 L3。
3. **长期记忆只有被检索并塞回窗口时才有用**。这就意味着，如果你的检索（Retrieval）没做好，硬盘里存了再多的长期记忆，AI 也看不到，相当于不存在。

**一句话总结你的问题**：上下文窗口是**“当前的工作台”**，长期记忆是**“仓库”**。仓库里的东西不搬上工作台就用不了，工作台上的东西不下单归档就会当垃圾扔掉。两者互不包含，靠“检索（Query）”和“存储（Index）”这两条传送带连接。

### 1.3长期记忆并不是知识库，但确实被存储在外部

#### 1.3.1什么是外部？

长期记忆绝对存储在“上下文窗口”之外。

- **本地外部**：你电脑硬盘上的 `SQLite.db` 文件（Hermes 的方式）或 `.lance` 数据目录。
- **云端外部**：远端的 PostgreSQL 或 LanceDB 云服务。

无论存哪里，它们都**不在**当前 AI 推理的那几十 K 显存里，**需要时得专门“查”和“取”**。

**长期记忆肯定需要专门“查”和“取”！**

#### 1.3.2不止是知识库，可以是知识库+个人经历（偏好）

| 类型                     | 像什么？                  | 对应你之前聊的组件                                         | 存储内容举例                                                 |
| :----------------------- | :------------------------ | :--------------------------------------------------------- | :----------------------------------------------------------- |
| **语义记忆（知识库）**   | 像**百科全书 / 代码文档** | 代码向量索引（Cursor）、项目 Wiki（OpenClaw）              | 函数签名、API 文档、README、代码片段、StackOverflow 问答。这是**静态的、公开的**。 |
| **情节记忆（个人经历）** | 像**私人日记 / 操作日志** | 会话历史（Hermes SQLite）、Hook 日志、Plans、Worktree 快照 | “上周三你改过这个配置”、“你重构 Auth 时踩过那个坑”、“你的编码习惯是 2 空格缩进”。这是**动态的、私有的**。 |

**产品致命误区**：很多 AI 产品只建了“知识库索引”（语义记忆），却忘了存“用户经历”（情节记忆）。结果就是：AI 虽然懂代码语法（知识），但永远记不住你刚才改到哪里了（经历），导致每次都要重新问一遍“我之前是怎么写的？”

#### 1.3.3为什么必须“储存在外部”？（工程上的硬性要求）

- **空间维度**：128K tokens 存不下几十万行的代码库，所以必须**剪裁、向量化、放外部**。
- **时间维度**：一次会话结束后（比如你关了 Cursor 或 5 分钟没发消息），**内存（上下文窗口）会被释放**。如果你没把关键结论存进外部的 SQLite 或 LanceDB，那这段对话就永久消失了。这就是为什么需要你上一轮提到的 **“写入门槛（Write Gate）”**——必须在外部落笔为安。

#### 1.3.4“外部”记忆的一个隐藏大坑（产品设计必知）

正因为记忆在外部，所以**它是有“滞后性”的**——这就回到了你之前纠结的 **“索引新鲜度”**。

- **知识库（代码库）更新**：你改了 `auth.js`，文件监听队列（File Watcher）检测到，触发异步索引更新（Cursor 的 30 秒陈旧度）。
- **个人经历更新**：你刚在对话里说“我不喜欢用箭头函数”，如果系统没立刻把这句话写入外部的 SQLite，等下一轮聊天时，AI 就忘了——因为这句话还在上一个上下文窗口里，但那个窗口已经被新对话挤掉了。

#### 1.3.5设计长期记忆系统

你可以这样定义你的长期记忆系统：

> **长期记忆 = 位于上下文窗口外部的持久化存储层。**
> 它由 **“静态知识索引”**（代码、文档的向量库）和 **“动态情节序列化”**（会话摘要、操作日志的 SQLite）共同构成。
>
> 它的核心任务不是“存得多”，而是**在下一次用户提问时，能通过检索（RAG）精准地把外部的那一小块数据搬回上下文窗口**。如果搬错了（检索失败），就算外部存了 1TB 的数据，当前的 AI 也等同于“失忆”。

所以，你现在的产品设计重点其实可以转移到：**“外部存什么”**（决定上限）和 **“外部怎么查”**（决定体验）。你之前关注的 LanceDB 解决前者，SQLite FTS5 解决后者。两者缺一不可。

> 可以理解是：模型的大脑“**短期记忆容量**”

## 四、Context Window有啥？

context window=系统指令（固定rules） + 历史摘要/快照（可被压缩） + 当前输入（触发词） + 实时生成的输出流（动态增长）

**超限的结果是被动裁剪或直接报错。**

## 五、为什么context window重要？

在 AI 应用开发里，Context Window 决定：

1. 能不能塞进整本书
2. 能不能分析完整代码仓库
3. 能不能长时间对话不“失忆”
4. [RAG](https://link.juejin.cn/?target=https%3A%2F%2Fblog.itpub.net%2F30126024%2Fviewspace-2112831) 能塞多少 chunk
5. [Agent](https://link.juejin.cn/?target=https%3A%2F%2Fdeveloper.aliyun.com%2Farticle%2F893805) 能带多少历史思考轨迹

## 七、疑惑

假设模型的最大上下文长度为**128K tokens**。在一次请求中，已有历史上下文 **120K tokens**，本轮输入 **8K tokens**，总长度已达到上限。

 若 **关键提示 A** 位于当前上下文的末尾 1K 区间内（最早的那批内容里），当模型开始自回归生成下一个 token 时：

已知128k = 历史 + 提问 + 回答 目前 历史+提问 === 128k

在最后一k区间里的提示是否仍然参与注意力计算

- **该提示 A 是否仍参与注意力计算？**
- **新生成的 token 是否会导致上下文长度超过 128K？如果超过，模型如何处理？**

### 7.1该提示 A 是否仍参与注意力计算？

**答案是：绝对会（100% 参与）。**

#### 7.1.1如果关键提示 A 位于最早（最开头）的那 1K 里

- **因果注意力（Causal Attention）的规则**：在自回归生成中，**新生成的 Token 可以看见（Attend to）它之前的所有 Token**，包括第 1 个到第 128K 个。
- **计算过程**：模型生成**第一个**新 Token（即第 128K+1 个 Token）时，它会读取内存中缓存的 **KV Cache（键值缓存）**。这个缓存里存着全部 128K 个历史 Token 的 Key 和 Value。无论提示 A 在开头还是结尾，只要它在缓存里，**新 Token 的 Q（查询向量）就会与 A 的 K（键向量）做点积计算**，计算出注意力分数。
- **物理位置的影响**：如果 A 在**开头（最早的 1K）**，由于绝对位置编码（如 RoPE）的存在，它与新 Token 的距离极远，注意力分数会受到**距离衰减**影响（分数可能较低），但依然会参与 softmax 归一化计算，影响最终结果。如果 A 在**末尾（输入结束前的 1K）**，那它的注意力分数会极高，影响力巨大。

#### 7.1.2如果关键提示 A 位于最后（最末尾）的那 1K 里

**物理位置的影响**：如果 A 在**末尾（输入结束前的 1K）**，那它的注意力分数会极高，影响力巨大。

### 7.2新生成的 Token 是否会导致上下文长度超过 128K？

**答案是：在数学逻辑上“会”，但在物理工程实现上“绝对不允许发生”。**

当模型开始生成第一个字时，当前序列长度逻辑上确实变成了 **128K + 1**。但大模型的推理引擎（如 vLLM、TensorRT-LLM、llama.cpp）有严格的物理显存限制。针对这种情况，工业界只有以下 **3 种处理方式**（按恶劣程度排序）：

#### 7.2.1方式 A（商业 API 的标准做法）：直接报错或截断

- **现象**：你调用 OpenAI / Claude / 国内大厂 API 时，如果 `Prompt_tokens + max_tokens > 128K`，API 会直接返回 `400 Bad Request` 错误（上下文长度超限），**不会生成任何新 Token**。
- **处理逻辑**：服务端会预先计算：`输入 128K + 预设输出 1K = 129K`，发现超限，直接拒绝推理。

#### 7.2.2方式 B（最粗暴的本地推理）：显存溢出（OOM）崩溃

- 如果本地运行开源模型，且框架没有做保护，当 KV Cache 试图分配 128K+1 的显存空间时，CUDA 会直接报 `Out of Memory`，程序崩溃退出。

#### 7.2.3方式 C（部分高级本地框架，如 llama.cpp 的“滚动窗口”）：物理丢弃最早的内容

- 针对你这种“输入刚好卡满”的情况，如果引擎设置了 **“滚动 KV 缓存（Rolling KV Cache）”** 或 **“上下文移位（Context Shift）”**，它会这样做：
  1. 检测到 `输入长度 == 最大限制`。
  2. **强制丢弃**最开头的 10%（即你提到的“最早的那批内容”——**关键提示 A 直接被删除了**）。
  3. 腾出 10% 的空间（约 12.8K），用来存放即将生成的新 Token。
  4. 此时，模型物理内存里只有 115K 的历史 + 新生成的 Token，**逻辑上下文窗口永远被锁死在 128K 以内**。

## 7.3与“记忆”直接相关的结论

1. **关于注意力**：**只要提示 A 在生成第一个字之前还在缓存里，它就一定参与计算。** 但当它参与完计算，生成第二个字时，如果引擎采用“方式 C（滚动窗口）”，提示 A 可能已经被物理移除了，导致后续生成完全失忆。
2. **关于超限**：**新 Token 不会导致长度“超过”128K，因为物理机制不答应。** 如果没报错，那一定是系统**主动“遗忘”了最早的一部分内容**（截断或滚动）来腾出空间。
3. 对设计产品的启发
   - 这就是为什么你必须**在请求之前**做 **“上下文压缩（Context Compression）”**。你不能把 120K 的历史原封不动地扔进去，因为这样 AI 就没有“输出空间”了。
   - 实战策略是：当历史达到 100K 时，后台必须**强制触发摘要（Summarization）**，把 100K 压成 5K 的“记忆碎片”，这样输入变成 `5K(摘要) + 8K(提问) = 13K`，留下 115K 的空间给 AI 自由发挥（写代码、生成文档），既保住了关键信息，又规避了“滚动窗口”暴力删数据的风险。这才是你产品架构里“压缩”模块存在的核心价值。
