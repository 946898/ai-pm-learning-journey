# LangGraph核心概念

LangGraph 是一个**有状态的工作流编排框架**，可以把它理解为“**带状态的流水线**”。它的核心由三部分组成：

## State状态/Node节点/Edge边

| 概念              | 是什么                                                       | 类比                                               |
| :---------------- | :----------------------------------------------------------- | :------------------------------------------------- |
| **State（状态）** | 一个共享的数据容器，在所有节点之间传递。通常是 `TypedDict` 或 Pydantic 模型 | 流水线上的**托盘**，装着所有需要的信息             |
| **Node（节点）**  | 一个 Python 函数，接收 State，做事情（调 LLM、查数据库），返回更新后的 State | 流水线上的**工位**，工人从托盘拿东西、加工、放回去 |
| **Edge（边）**    | 连接节点的箭头。**普通边**是“A 之后执行 B”；**条件边**是“A 之后根据 State 的值决定走 B 还是 C” | 流水线上的**传送带**，决定下一步去哪               |

**节点完成工作，边指示下一步该做什么。**

## 边学边做

### 为什么要激活虚拟环境？

**核心原因：隔离依赖包，防止“版本冲突“和”权限污染“。**

- **不激活（全局环境）**：你执行 `pip install`，会把包装到系统的 Python 目录里（比如 `C:\Python39\Lib\site-packages`）。如果你同时做 A、B 两个项目，一个要 `langgraph==0.1`，一个要 `langgraph==0.5`，全局安装就会互相覆盖，导致项目 A 突然跑崩。
- **激活后（虚拟环境）**：你执行 `pip install`，只会装到当前项目下的 `venv_ai\Lib\site-packages` 里。每个项目有自己的“独立包仓库”，互不干扰。

**通俗比喻**：

- 不激活 = 所有工具都扔在一个大工具箱里，找东西费劲，还可能拿错。
- 激活 = 每个项目有一个专属小工具箱，要用时“打开”它，里面的工具只属于这个项目。

### 为什么区分Graph API和Functional API？

LangGraph 的 Graph API 和 Functional API 是构建智能体工作流的两种不同方式，它们在**设计哲学、控制流、状态管理**等方面有显著区别。简单来说：

- **Graph API** 是**声明式**的，让你像画流程图一样，用**节点（Node）** 和**边（Edge）** 来定义工作流。
- **Functional API** 是**指令式**的，更像写普通的 Python 函数，用 `@entrypoint` 和 `@task` 装饰器来组织逻辑。

#### 两种 API 的详细对比

下面的表格可以帮你更直观地理解它们的区别：

| 特性         | Graph API (图 API)                                           | Functional API (函数式 API)                                |
| :----------- | :----------------------------------------------------------- | :--------------------------------------------------------- |
| **设计哲学** | **声明式 (Declarative)**：将工作流定义为有向图。             | **指令式 (Imperative)**：使用标准 Python 结构定义工作流。  |
| **控制流**   | 通过**显式**的**节点**和**边**来控制。                       | 使用 `if`、`for`、`函数调用`等**标准 Python 语法**。       |
| **状态管理** | 需要**显式声明**一个 `State` 对象，并定义 `reducers` 来管理状态更新。 | **无需显式状态管理**，状态隐式地通过函数参数和返回值传递。 |
| **可视化**   | **支持**，可以方便地将工作流可视化为图表，便于调试和沟通。   | **不支持**，因为图结构是在运行时动态生成的。               |
| **使用场景** | 适合**复杂决策树、多分支逻辑、并行处理**等需要精细控制的场景。 | 适合**快速原型开发、线性流程、迁移现有代码**等场景。       |

> 注意：两种 API 共享相同的底层运行时，可以在同一个项目中混合使用。

#### 各自的应用场景

根据上面的对比，这两种 API 有各自最适用的场景。

#### 何时使用 Graph API？

- **复杂的决策流程**：当你的工作流有多个条件分支和决策点，用图的方式来定义会更清晰、更易于管理。
- **需要精细的状态控制**：当不同节点需要共享和修改复杂的状态时，Graph API 的显式状态管理会更合适。
- **并行处理**：当需要并行执行多个任务，并在之后合并结果时，Graph API 可以自然地进行处理。
- **团队协作与可视化**：图形化的表示方式有助于团队成员理解、讨论和维护复杂的工作流。

#### 何时使用 Functional API？

- **快速原型开发**：如果你希望快速验证一个想法，避免定义图结构的“仪式感”，Functional API 是更轻量的选择。
- **线性或简单分支的工作流**：如果你的工作流主要是顺序执行，或只有简单的 `if/else` 逻辑，用 Functional API 会更直接。
- **为现有代码添加 LangGraph 能力**：这是 Functional API 的一个核心设计目标。如果你已有一些 Python 函数，想为它们增加**持久化、记忆、人机协同**等能力，用 `@task` 和 `@entrypoint` 装饰器可以最小程度地改动代码。

#### 总结与建议

这两种 API 并非互斥，而是针对不同场景的互补工具。

结合你正在开发的**采购订单审核 Agent**，这里有一个具体建议：

- **核心工作流**：如果你的审核流程包含**多个步骤**（解析订单、风险分析、供应商核验、决策），且未来可能有**复杂的分支逻辑**（如根据风险等级分流），使用 **Graph API** 会让整个流程的结构更清晰，也更容易调试和维护。
- **特定功能模块**：如果你想把某个特定功能（如“供应商核验”）模块化，并方便地在不同项目间复用，可以考虑用 **Functional API** 将其封装成一个独立的 `@task`。

简而言之，**需要精细控制和清晰结构时选 Graph API，需要快速开发和代码复用性时选 Functional API**。
