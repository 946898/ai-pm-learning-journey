## 一、为什么Agent评测需要体系化？

> Agent 评测是把「不稳定的智能行为」持续收敛成「可发布的工程质量」，**不是上线前抽查**。

**Agent相比传统软件存在三大特性：非确定性（同输入不同输出）、黑盒化（决策过程不透明）、错误级联放大（小错被后续步骤放大）**。仅靠少量Case测试无法保障稳定性，“跑通一次”不代表可用（你自己设计的多人协同标书辅助撰写装配台Agent就是如此，应该要深有体会！！！），模型/Prompt/工具参数的微小改动都可能导致原有场景失效，甚至存在“结果正确但过程违规”的假阳性风险。

- 同一Prompt跑通一次，不代表稳定可用。
- 每次改模型、改Prompt或改工具参数，都可能把原来好用的场景改坏，而且不容易第一时间发现。
- 多步链路里前面有一个小偏差，也可能在后面被放大成错误结论。
- 更隐蔽的是，有些样本最终答案看起来对，但执行路径已经偏离或带风险，这类假阳性（结果看起来通过、过程其实有问题）在生产里迟早会暴露。

**评测体系必须嵌入研发流程，而不是上线前人工抽查。**评测平台不只是“出分工具”，**它的核心价值是把问题稳定转化为可执行修复，形成持续迭代闭环**。

一套成熟的Agent评测体系至少要回答三类问题：**①能力水位（任务完成率、幻觉率等基线指标）；②变更风险（新旧版本差异、场景退化情况）；③优化方向（失败根因、责任归属、修复动作）**。

## 二、按Agent类型差异化设计评测

> 先分清Agent类型，再定义评测指标；类型不分，评测结果基本不可用。

不同类型Agent的评测重点完全不同，通用指标会导致结果失效：

![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\640.png)

评测底层骨架可以复用：执行用例、采集 Trace（执行轨迹）、运行 Scorer（评分器）、生成报告、做根因归类。**真正需要按业务定制的是“评什么”和“怎么判”**。

此外需单独评测Agent的Skill能力：触发准确性、参数正确性、关键步骤合规性、异常容错能力、产物可用性。

## 三、对话Agent的特殊评测逻辑

> 对话Agent不能只看单轮答案，必须把整段会话是否解决问题作为主评判对象

对话类Agent不能只看单轮答案，需以“整段会话是否解决问题”为核心，正确做法是同时看历史 Turn（单轮）、Session（整段会话）、Trace（执行轨迹）、Outcome（最终结果）四个层次。

需重点关注5个特殊难点：![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\微信图片_2026-07-23_095839_921.png)

- 关于业务流程约束，我理解是传统PRD里面的业务流程、逻辑约束，比如在XXX状态下不允许结算，不允许发货等等。

- 制定情绪回应评分标准+检查Agent输出+人工抽样校准来验证Agent质量
- 转人工的标准、触发率、时机、交接摘要准确率（方便人工快速进入状态）

对于任务执行型，以及客服、营销、导购、售后这类对话型 Agent，要**特别重视“多次运行的一致性”**。**同一个任务建议重复运行多次，观察两类结果：**

- **至少一次成功率（MVP）：**同一个任务跑N次，只要有一次成功，就说明 Agent 具备完成这个任务的可能性，适合观察能力上限；
- **连续成功率（稳定版本）**：同一个任务跑N次，必须每次都成功，才说明 Agent 稳定可靠，适合客服、支付、退款、合规等生产级场景。

在版本对比场景里，至少一次成功率和连续成功率还需要配套统计检验，避免把随机波动误判为能力变化。建议在报告中同时给出：

- 关键指标的置信区间：说明结果的稳定范围，避免只看单点分数；
- 与基线版本的显著性判断：判断差异是“真实提升/下降”还是随机波动；
- 最小可感知变化阈值：提前定义“至少提升（或下降）多少才算值得做发布判断”。

## 五、评测数据集建设

> 评测集并非线上数据的随机抽样，是围绕高风险路径、关键逻辑和失败模式设计出来的质量资产。

评测集不是线上数据的随机抽样，而是围绕高风险路径、关键逻辑、失败模式设计的“质量资产”，核心构成：

1. **专家设计用例**：覆盖核心流程、高风险边界，锚定业务共识；
2. **扩展用例**：基于专家用例，用规则确定结构字段，LLM生成自然语言表达，扩充长尾、异常、对抗场景；
3. **线上真实数据**：按场景/风险分层抽样，避免幸存者偏差；
4. **badcase回流**：线上失败、人工质检问题沉淀为用例，避免同类问题复发。 初期先建50-200条高质量“黄金评测集”作为基线，成熟后再扩展为分类采样集、长尾集、对抗集。

**推荐的数据集建设路径：**

评测集至少包括四类来源：

![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\微信图片_2026-07-23_193103_352.png)

- **专家设计用例负责“定标准”，扩展用例负责“扩覆盖”**。扩展用例不应脱离专家用例和业务规则自由生成，**建议先用规则确定场景类型、订单状态、用户诉求、期望动作等结构字段**，再用 LLM 生成更接近真实用户的自然语言表达。
  - 主流程-专家设计用例
  - 分支流程-扩展用例
- **先做** 50-200 条高质量 **golden set**，覆盖核心业务路径和高风险边界；**成熟后再扩展到分类采样集、长尾集、对抗集、线上回流集**。
- 对包含 Skill 的 Agent，用例设计可沿“触发、核心逻辑、产物质量、异常容错”四类组织：
  - 先确认该不该触发 -> 再确认触发后过程对不对 -> 再检查最终产物好不好 -> 最后验证异常输入、工具失败、边界条件下能不能稳住。
  - 核心逻辑用例通常最多，应该覆盖主要分支路径和高风险分支。
- 示例
  - 专家设计用例可以是“用户要求退款但订单已发货，Agent 必须先查订单状态再解释规则”；
  - 扩展用例可以是在同一退款场景下生成不同用户说法、不同订单状态和不同情绪强度；
  - 线上真实数据可以来自真实客服会话或工单记录；
  - badcase 回流可以来自“Agent 直接承诺一定退款成功”这类线上Badcase。

## 六、三层评分体系

> 可以用规则来主判打分，LLM Judge主要用于语义和策略这类非确定性维度。

优先用规则判“硬条件”，LLM Judge判“语义/策略合理性”，人工仅做终判与校准，评分器可以分三类，优先级从高到低：

![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\微信图片_2026-07-23_193630_927.png)

1. **整体原则是：**

   a. 规则看“硬条件是否满足”（工具、状态、字段、禁用动作），LLM看“语义/策略是否合理”（解释质量、策略妥当性等）；

   b. 能写成代码的项由规则主判；

   c. 在高风险、抽检、低置信或变更观察期样本上，可以追加LLM-as-Judge；

   d. 能由规则和Judge稳定判断的，不长期依赖人工；人工主要用于确认业务标准、处理冲突和高风险终判。

2. **LLM-as-Judge不能只写“请从1到5打分”，好的Judge至少需要：**

   a. 明确评分标准，每个分档有可执行标准；

   b. 输出reason，方便定位和后续badcase聚类；

   c. few-shot示例，尤其要包含边界样本和判定的COT逻辑；

   d. 周期性校准，Judge与人工复核一致率达到约85%（参考值）后再进入日常自动化；

   e. 偏差治理（如：评测模型和生成答案的模型是同一个或同系列，它可能更喜欢自己风格的回答），可以引入多个不同的LLM进行对抗打分评测。

3. **人工评分不适合承担日常全量打分，更适合用在以下几个环节：**

   a. 新建评测集时，确认样本的期望行为和打分标准。例如订单已发货、用户拒收、商家参与运费险且商品属于特殊类目时，Agent 应该解释退款路径、建议拒收、转人工，还是先发起售后单，需要业务先定口径；

   b. 校准 LLM-as-Judge，抽取一批样本由人工先判。例如 Judge 能判断回复是否礼貌、是否解释清楚，但无法可靠判断“叠加优惠后的差价是否应该补偿”这类依赖内部政策和实时活动规则的问题；

   c. 处理争议样本；

   d. 抽查高风险场景。例如赔付、隐私、合规、投诉升级等样本，即使自动评分通过，也要定期人工复核。

4. **是否进入人工评分不应只由LLM Judge的分数阈值决定，而应做成“人工评分路由”：**

   a. 自动评分不确定时进入人工，例如 Judge 分数落在通过/不通过边界附近、置信度低、reason 含糊，或多个 Judge 结论分歧明显；

   b. 新模型、新 Prompt、新工具 Schema（字段结构与校验规则）、新业务活动上线时，抽样进入人工，用于发现自动评分盲区并校准 Judge；

   c. 规则与 LLM-as-Judge 结论冲突时进入人工，由人工做终判和口径回收。

5. **对于 Skill 密集型任务，可以针对 Skill 做专项评测，避免只看端到端结果。这里的 Skill 检查项用于“判分与分流”，不用于“归因与定责”；归因和责任判定统一放到第 7 章的 badcase 分析阶段：**

![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\微信图片_2026-07-23_195020_800.png)

6. **持续评测，可以维护一套 Judge 金标校准集，并在每次 Judge Prompt/模型版本升级时固定产出三类统计指标：**

1. a. 与人工标注一致性（例如和人工标注的一致率或多个大模型对抗打分的一致性 -> Kappa 系数）；

2. b. 高风险样本漏判率（尤其是合规、资损、隐私场景）；

3. c. 边界样本稳定性（同批样本重复评测时的波动幅度）。

7. **在评分的实际执行阶段，可以采用“分层筛查”识别 badcase，并控制进入 RCA 的样本质量：**

用低成本、确定性强的能力快速分流，再对疑似样本做更完整的语义和策略判断，最后把高风险、低置信或冲突样本交给人工终判。

评分结果不应只输出 **pass/fail**，还应输出初始问题分类、问题现象、置信度和判定依据。这样第 7 章做根因定位时，才能基于“现象入口”收敛候选模块，而不是对所有模块无差别排查。这里的问题分类是现象层分类，例如“事实性错误”“答非所问”“过度承诺”，不等同于最终根因或责任模块。

​	a. 粗筛层主要使用规则 Scorer，必要时配合轻量级 LLM-as-Judge，用于快速识别明显通过、明显失败和存疑样本；

​	b. 精判层使用完整规则和 LLM-as-Judge，加载更完整的评分标准，确认样本是否为 badcase，并产出初始问题分类、问题现象、置信度和判定依据；

​	c. 人工复核层使用 Human Scorer，处理高风险、低置信、规则与 Judge 结论冲突、业务口径未固化的样本。

![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\微信图片_2026-07-23_195507_755.png)

对结果假阳性case（*最终回复看起来正确，但执行路径或风险检查不通过*）可增加过程型和风险型检查：

![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\微信图片_2026-07-23_195557_277.png)

综上，除了输出分值、pass/fail，**评分结还应输出问题分类（*可以提前由产品、业务、研发协商定义出来*）、问题现象、置信度和判定依据**。这样第 7 章做根因定位时，才能基于“现象入口”收敛候选模块，而不是对所有模块无差别排查。这里的问题分类是现象层分类，例如“事实性错误”“答非所问”“过度承诺”，不等同于最终根因或责任模块。





## 七、Badcase根因定位（RCA）

> Badcase根因定位的核心是把错例稳定追到责任模块和可修复原因。
>
> Tips：Badcase 不只来自离线评测集，也来自线上会话、人工质检、投诉工单、低满意度样本、版本回归失败和监控告警（可以作为评测集的补充）。在进入根因定位前，我们把入口统一为两条通路：

![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\微信图片_2026-07-23_195926_262.png)

*这两条链路最终应汇入同一个 badcase 任务表和同一套根因分析 Pipeline（处理流水线），这样既能覆盖线上长尾，也能支持运营和研发主动排查。*



Badcase RCA（根因分析）可以用一个“先收集证据、再收敛范围、最后定责落盘”的通用链路来表达：

![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\微信图片_2026-07-23_200027_370.png)

1. **证据汇总**：通过traceId等关联全链路日志、模块输入输出、工具调用记录；
2. **范围收敛**：基于“问题现象×功能模块”映射表缩小排查范围（如“事实性错误”优先排查检索、知识筛选模块），未命中场景用LLM兜底；
3. **分模块诊断**：逐个排查候选模块，输出问题摘要、关键证据、改进建议；
4. **责任判定**：结合规则、模块结论、根因知识库，明确主责/次责模块、问题分类、问题枚举、修复建议；
5. **结构化落盘**：结果写入任务记录，支持聚类分析（如“已发货退款场景23次跳过订单校验”），沉淀可统计的根因标签体系（覆盖意图识别、上下文记忆、检索召回、工具执行、流程合规等9类常见问题）。 落地采用“确定性优先、模型辅助、人审兜底”策略，高风险/低置信样本必须人工复核。

## 八、自动优化建议产出

> 优化建议要落到明确owner、修复动作和回归验证，不能只停留在报告结论里。

优化建议需落地为可执行的工单，避免泛化表述，结构化行动项需包含：问题摘要、影响范围、根因判断、证据、建议动作、Owner、验收标准、优先级。

按根因匹配优化手段：Prompt/策略问题改系统提示、工具说明；RAG问题补知识、优化召回逻辑；工具问题优化Schema、降级策略；业务流程问题显式化SOP为状态机；产品交互问题增加澄清、人工接管入口；模型能力问题考虑换模、微调。

优化等级可分为：L0仅报告问题、L1自动生成工单、L2生成可审阅的配置建议、L3自动生成待人工审核的修复PR。

## 九、全链路闭环

> 评测平台的终点不是报告，而是把线上失败持续转化为可复用的研发资产。

一套完整评测系统不能停在报告和建议上，还要把 Badcase 变成 bug、工单、配置变更、知识库修订、训练数据和回归用例，并在每个版本周期里持续运行。

一个完整的 Agent 评测平台通常包含九个能力模块：

![微信图片_2026-07-23_093916_930](D:\工作集合\1zhouxu48\2.9-AI员工专项\2.9.0-【AI先锋】\1-自我提升\ai-pm-learning-journey-main\images\微信图片_2026-07-23_201031_475.png)

**评测的终点是“线上失败持续转化为研发资产”，需打通离线评测与线上反馈：**

- 发布阶段联动离线门禁与线上灰度，跟踪离线质量、线上体验、业务结果三类信号，若线上指标恶化立即回滚；
- 将Badcase转化为三类资产：回归用例（防复发）、Prompt/工具改进建议（修当前问题）、业务规则/知识库修订（修源头问题）；
- 沉淀三条生产线：评测生产线（用例库、回归集）、运营生产线（知识/SOP/规则缺口）、模型生产线（微调样本、偏好数据）；
- 建立样本入库标准（可复现、期望行为明确、根因清晰、有代表性、已脱敏），定期治理回归集，避免无限膨胀。
  - 失败可复现，或至少有稳定 Trace 和人工确认；
  - 期望行为明确，可以写成规则、Judge 标准或人工验收标准；
  - 根因标签清楚，能归到具体能力域或业务流程；
  - 样本有代表性，能覆盖一类问题，而不是一次性环境抖动；
  - 已完成脱敏，符合隐私和数据合规要求。

最终，评测平台要沉淀的是一套**可复用的质量资产库**：用例库、Trace 库、根因标签库、修复建议库、Judge 校准集和回归集。**质量资产越厚，Agent 迭代越不会靠个人经验和临时救火。**

## 十、落地核心原则

> 评测体系的成败，取决于它能否在跨团队协作中长期、稳定地产生可执行行动项。

评测体系的成败取决于能否长期稳定产出可执行行动项，避免“分数好看但无实际优化”，最终沉淀为可复用的质量资产库（用例库、Trace库、根因标签库、修复建议库等），降低后续迭代成本，摆脱对个人经验的依赖。
