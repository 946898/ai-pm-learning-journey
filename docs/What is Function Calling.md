## 一、Conclude

1. 模型的能力仅限于“回答”,不具备调用本地函数做出额外动作的能力，**Function Calling的主要作用是赋予大模型调用本地函数的能力**
2. 和Agent的关系，**Agent 底层通常就是基于 Function Calling 实现的**
3. 和prompt的区别是：**prompt是 让模型“尽量按格式输出” ，cunction calling则是模型被强制输出结构化格式，区别在于稳定些**
4. 和MCP的区别： **Function Calling = 模型调用本地函数，MCP = 标准化“外部工具调用协议”**

## 二、什么是Function Calling？

> 👉 让大模型在对话过程中，**按照你预定义的函数结构，自动生成可执行的函数参数 JSON**，由你的程序去真正执行逻辑。

模型负责“决定调用什么函数 + 生成参数”，
你的程序负责“真正执行函数”。

## 三、为什么会有Function Calling？

原始大模型【只能输出文本】（基于训练数据）

比如问：

> 帮我查一下杭州属于哪个省

会基于训练数据答复：

> 杭州属于浙江省

如果问：

> 杭州明天天气怎么样

只能答（训练数据不包含时）：

> 不知道/或者编一个答案给你

因为这组数据，他的训练数据里面没有，天气这种数据，一般都是实时产生的，不能作为固定数据提前训练进去。试试天气，一般是一些服务商提供查询api，才能获取到。**而大模型本身，并不具备主动调用查询api的能力，或者说，它的能力仅限与“回答”**

核心问题是：**怎么让模型主动调用你系统里的真实函数？**

Function Calling就是解决这个问题的可行办法。

### 3.1工作流程

### 3.1.1定义函数

重点，这个函数是你在你自己的系统里定义的，不是模型里面内置的

### 3.1.2用户输入

今天杭州天气怎么样？

### 3.1.3模型回答

#### 3.1.3.1模型处理

假设提示词有：

> 如果检测到输入有提问天气情况的，请按如下xxx格式给他返回，其中function_call.name字段是为字符串：getWeather,function_call.arguments.city为输入中询问的城市名称，如果没有城市，默认返回xxxxx

#### 3.1.3.2生成参数

```json
{
  "function_call": {
    "name": "getWeather",
    "arguments": {
      "city": "北京"
    }
  }
}
```

### 3.1.4程序处理

#### 3.1.4.1识别到固定格式

```java
if(function_call.name === 'getWeather'){
    this.getWeather(function_call.arguments.city)
}
```

#### 3.1.4.3拿到真实天气数据

## 四、Function Calling核心本质

> 让模型输出【结构化JSON】，而不是自然语言。

模型不是直接执行函数，【它只是一个“决策器+参数生成器”】

## 五、和Agent的关系

- Function Calling = 调用1个函数
- Agent = 根据任务多步调用函数 + 规划步骤

Agent底层通常就是基于Function Calling实现的

## 六、和MCP的关系

- Function Calling = 模型调用本地函数
- MCP = 标准化“外部工具调用协议”

MCP可以看作是Function Calling 的协议升级版本 + 工具生态标准化
